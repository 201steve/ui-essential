# 2. new Set()은 중복되지 않는 고유값을 다뤄야 할 때 유용하다.
* 동일한 값을 두번 저장하지않는다.

* has() 메서드로 특정 값으 존재 여부를 배열보다 빠르게 확인할 수 있다.

# 3. ARIA
원칙 1

role은 이름표가 아니라, 그 역할에 맞는 키보드/상태/동작까지 제공하겠다는 약속이다.

원칙 2

ARIA는 기존 HTML 의미를 더 좋게 보강할 수도 있지만, 잘못 쓰면 원래 의미를 덮어써서 접근성을 망칠 수도 있다.

전체 결론

가능하면 네이티브 HTML을 먼저 쓰고, ARIA는 정말 필요한 의미와 상태만 신중하게 보강하는 용도로 써야 한다.

## 3-1 aria-hidden
이 속성이 있으면 접근성 트리에서 제외된다.즉, 시각 사용자는 아이콘을 볼 수 있는데
스크린 리더 사용자는 이 span이 없는 것처럼 처리된다. 

단순히 화면에서 안보이게 만들거나, 레이아웃에서 제거하는게 아니라
보조기술이 이 요소를 인식하지 않게 하는것.



왜 이렇게 사용하냐면,
```
<button>
배송정보
<span>▼</span>
</button>
```
이렇게 되면 스크린리더 사용자는 span 정보까지 듣게 된다. span은 단순 장식용이지 기능을 하는게 아니기때문에
환경에 따라 불필요한 정보가 섞일 수 있어서 aria-hidden으로 접근성 트리에서 제외하는게 맞다.
## 3-2 aria-expended
이 속성이 있으면 보조기술 사용자는 열려있는지 닫혀있는지 상태를 듣게된다.

# test
## describe
```jsx
describe("대상",()=>{})
```
"대상"에 대한 테스트들()=>{}이라는 그룹명을 붙여놓은것.

대상에 대한 테스트가 하나일수도 있고, 여러개 일 수도 있는데, 나중에 테스트가 많아지면
테스트 전부 describe('대상")에 넣게 된다.

## it
it은 describe한 그룹에서 개별 테스트 1개를 정의하는것.
```jsx
it("기대하는 것",()=>{})
```
예를들어

```jsx
describe("Accordion",()=>{
    it("섹션 제목들을 렌더한다",()=>{
        ...
    })
})
```
이렇게 되면 Accordion은 섹션 제목들을 렌더해야 한다 라는 테스트 시나리오를 적은것.
테스트 이름은 아주 중요하다. 왜냐면 실패했을 때 이 문장이 그대로 출력되기 때문.

그래서 테스트 이름은 보통 무엇을 기대하는지 가 드러나게 쓰는게 좋다.
##  render
``` jsx
import {render,screen} from '@testing-library/react'
```
render는
React 컴포넌트를 테스트용 DOM(JSDOM)에 그려주는 함수

## screen
screen은
렌더링된 화면에서 요소를 찾을 때 쓰는 전역 도우미
예를들어:
* 버튼 찾기
* 텍스트 찾기
* input 찾기

등을 할 수 있다

screen.getByRole('button',{name:/html/i}) 이렇게 하면, 화면에서 이름이 html인 버튼을 찾아라.

## expect
expect는 결과 검증하는 메서드
```jsx
expect(screen.getByRole("button",{name:/html/i})).toBeInTheDocument()
```

기대한다. 화면에서 button에 이름이 html인 역할을 가진요소가 현재 DOM 문서 안에 존재한다.
